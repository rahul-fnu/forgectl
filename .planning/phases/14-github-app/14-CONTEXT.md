# Phase 14: GitHub App - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Users interact with forgectl entirely through GitHub — triggering runs via webhooks, issuing slash commands, approving/rejecting via reactions, asking clarification questions mid-run, and reviewing results with check runs and auto-generated PR descriptions. No new agent capabilities or workflow features — this phase is the GitHub interaction surface on top of existing orchestration, governance, and durability.

</domain>

<decisions>
## Implementation Decisions

### Webhook routing & triggers
- Webhook receiver is a new Fastify route on the existing daemon (`/api/v1/github/webhook`), not a separate process
- HMAC-SHA256 verification on every incoming webhook using configured secret
- Trigger events: issue labeled with configured label (e.g. `forgectl`), issue opened with that label, issue comment containing slash command
- Webhooks coexist with polling — webhooks for instant response, polling as safety net for missed webhooks. Dispatcher deduplicates (if run already exists for issue, skip)
- GitHub App credentials configured in `forgectl.yaml` — new `github_app` section with `app_id`, `private_key_path`, `webhook_secret`. Private key stays on disk. Installation ID auto-discovered or configured per repo

### Slash command design
- Command prefix: `/forgectl` (full name, no ambiguity)
- Full command set: `/forgectl run [workflow]`, `/forgectl rerun`, `/forgectl stop`, `/forgectl status`, `/forgectl approve`, `/forgectl reject`, `/forgectl help`
- `/forgectl run` accepts workflow name as only argument — no flag overrides. Other config comes from WORKFLOW.md
- Invalid or unauthorized commands: add :x: reaction to the command comment, then post a reply explaining the error (unknown command, no permission, invalid args)

### Bot comment style
- Run results use collapsible `<details>` sections: summary line at top (status + duration + cost), then expandable sections for changes, validation results, cost breakdown
- Progress updates: post one "Run started" comment, then edit it as phases complete (validation attempts, retries, output collection). Final edit shows results. One comment total per run
- Clarification questions: bot posts comment @mentioning the issue author with the question. Run pauses via existing DURA-03 (`waiting_for_input`). Resume triggered when user replies (reply webhook)
- Auto-generated PR descriptions: full summary with changes list, validation results, cost breakdown, linked issue (`Closes #N`), agent/workflow used, "Generated by forgectl" footer

### Reaction mappings
- :+1: → Approve (`pending_approval` → `running` or `pending_output_approval` → `completed`)
- :-1: → Reject (`pending_approval` → `rejected`)
- :rocket: → Trigger run (equivalent to `/forgectl run`)
- :arrows_counterclockwise: → Rerun (re-dispatch same issue)
- :eyes: → Bot acknowledgment (bot adds this to confirm receipt of any command or reaction)
- Reaction scope: recognized on bot comments AND issue body. Not on arbitrary user comments (prevents accidental triggers from casual thumbs-up)
- Permission check: GitHub API collaborator check — only users with write or admin access can trigger actions. Non-collaborators' reactions and commands are silently ignored

</decisions>

<specifics>
## Specific Ideas

- Progress comment should show a checklist that gets checked off as stages complete (agent executing, validation attempt N, output collection)
- Clarification comment should quote the question and include "Reply to this comment to continue the run" with italic note "Run paused • will resume when you reply"
- Bot should feel like a teammate, not a CI system — conversational but structured

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/tracker/github.ts`: Existing GitHub adapter with `postComment()`, `updateLabels()`, `updateState()` — raw fetch against GitHub API. GitHub App replaces token auth with app installation auth
- `src/governance/approval.ts`: `approveRun()`, `rejectRun()`, `requestRevision()` — reaction handlers wire directly to these
- `src/durability/pause.ts`: `pauseRun()`, `resumeRun()` with `PauseContext` — clarification flow uses this for mid-run pauses
- `src/daemon/routes.ts`: Existing Fastify route registration pattern with `RouteServices` dependency injection
- `src/logging/events.ts`: `emitRunEvent()` for tracking webhook/command events in audit trail
- `src/orchestrator/dispatcher.ts`: `dispatchIssue()` — webhook handler calls this to trigger runs

### Established Patterns
- REST API routes: Fastify routes with standard `{ error: { code, message } }` envelope for errors
- Run status transitions: `runRepo.updateStatus()` with status field on runs table
- Fire-and-forget dispatch: `void executeWorkerAndHandle()` in dispatcher — webhook handler follows same pattern
- Config merge: 4-layer priority — GitHub App config goes in forgectl.yaml layer

### Integration Points
- `src/daemon/server.ts`: Register webhook route alongside existing routes
- `src/daemon/routes.ts`: Add GitHub App routes (webhook endpoint)
- `src/tracker/github.ts`: May need to support both token auth (polling) and app installation auth (webhook responses)
- `src/config/schema.ts`: Add `github_app` section to config schema with zod validation
- `src/orchestrator/dispatcher.ts`: Webhook handler calls `dispatchIssue()` with TrackerIssue from webhook payload

</code_context>
