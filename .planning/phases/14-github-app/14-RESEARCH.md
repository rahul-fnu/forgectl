# Phase 14: GitHub App - Research

**Researched:** 2026-03-10
**Domain:** GitHub App integration (webhooks, slash commands, check runs, reactions)
**Confidence:** HIGH

## Summary

Phase 14 adds a GitHub App interaction layer on top of forgectl's existing orchestration, governance, and durability subsystems. The core work is: (1) a Fastify webhook endpoint with HMAC-SHA256 verification, (2) event routing for issues/comments/reactions, (3) slash command parsing, (4) permission checks via GitHub collaborator API, (5) bot comment management with progress updates, (6) check run lifecycle on PRs, and (7) auto-generated PR descriptions.

The Octokit ecosystem provides well-typed libraries for every piece. `@octokit/app` handles authentication (JWT for app, installation tokens per repo). `@octokit/webhooks` handles signature verification and typed event dispatch. The installation Octokit instance (from `@octokit/app`) provides authenticated REST API calls for comments, reactions, check runs, and collaborator checks. No custom crypto or HTTP client code is needed.

**Primary recommendation:** Use `@octokit/app` as the single entry point. It bundles webhook verification, installation auth, and REST client. Wire its `webhooks.on()` handlers into a Fastify route (not `createNodeMiddleware`, which assumes Express/Node HTTP). Keep all GitHub-specific code in a new `src/github/` module, separate from the existing `src/tracker/github.ts` polling adapter.

### Pattern 1: Fastify Webhook Route with Raw Body
**What:** Fastify parses JSON by default, but @octokit/webhooks needs the raw string body for HMAC verification. Use Fastify's `addContentTypeParser` to preserve raw body.
**When to use:** Always -- this is the webhook entry point.
**Example:**
```typescript
// Source: @octokit/webhooks docs + Fastify raw body pattern
app.addContentTypeParser(
  'application/json',
  { parseAs: 'string' },
  (_req, body, done) => { done(null, body); }
);

app.post('/api/v1/github/webhook', async (request, reply) => {
  const id = request.headers['x-github-delivery'] as string;
  const name = request.headers['x-github-event'] as string;
  const signature = request.headers['x-hub-signature-256'] as string;
  const payload = request.body as string; // raw string for HMAC

  await webhooks.verifyAndReceive({ id, name, payload, signature });
  reply.code(200).send({ ok: true });
});
```

**CRITICAL DETAIL:** The webhook route MUST be registered on a separate Fastify instance OR use a route-scoped content type parser to avoid breaking JSON parsing for all other routes. Recommended approach: register the webhook route with `{ config: { rawBody: true } }` or use Fastify's `@fastify/raw-body` plugin scoped to the webhook prefix.

### Pattern 2: Installation Octokit per Event
**What:** Each webhook event provides the installation ID. Use `app.getInstallationOctokit(installationId)` to get an authenticated client scoped to that installation.
**When to use:** Every webhook handler that needs to call GitHub APIs.
**Example:**
```typescript
// Source: @octokit/app docs
githubApp.webhooks.on('issue_comment.created', async ({ octokit, payload }) => {
  // octokit is already authenticated for this installation
  await octokit.rest.issues.createComment({
    owner: payload.repository.owner.login,
    repo: payload.repository.name,
    issue_number: payload.issue.number,
    body: 'Hello from forgectl!',
  });
});
```

Note: When using `app.webhooks.on()`, the handler receives `{ octokit, payload }` where `octokit` is already authenticated for the installation. No manual `getInstallationOctokit()` call needed inside event handlers.

### Pattern 3: Edit-in-Place Progress Comment
**What:** Create one comment at run start, store the comment ID, update it as the run progresses.
**When to use:** GHAP-03 progress tracking.
**Example:**
```typescript
// Create initial comment, store ID
const { data } = await octokit.rest.issues.createComment({
  owner, repo, issue_number,
  body: buildProgressComment({ status: 'started', checklist: [] }),
});
const commentId = data.id;

// Later, update in-place
await octokit.rest.issues.updateComment({
  owner, repo, comment_id: commentId,
  body: buildProgressComment({ status: 'validating', checklist: ['agent', 'validation-1'] }),
});
```

### Pattern 4: Mapping Reactions to Runs
**What:** When a reaction event fires on a bot comment, look up the run ID associated with that comment.
**When to use:** GHAP-07 reaction-based approvals.
**Recommendation:** Store a mapping of `commentId -> runId` in the runs table (new column or separate table). When the bot creates a progress comment, record the mapping. When a reaction arrives on that comment, look up the run.

### Pattern 5: Slash Command Parser
**What:** Simple regex-based parser for `/forgectl <command> [args]`.
**When to use:** GHAP-04.
**Example:**
```typescript
interface ParsedCommand {
  command: string;       // 'run' | 'rerun' | 'stop' | 'status' | 'approve' | 'reject' | 'help'
  args: string[];        // e.g., ['my-workflow'] for /forgectl run my-workflow
}

function parseSlashCommand(body: string): ParsedCommand | null {
  const match = body.match(/^\/forgectl\s+(\w+)(?:\s+(.+))?$/m);
  if (!match) return null;
  return {
    command: match[1],
    args: match[2]?.split(/\s+/) ?? [],
  };
}
```

### Anti-Patterns to Avoid
- **Using createNodeMiddleware with Fastify:** It assumes Node's `http.IncomingMessage`/`http.ServerResponse`. Fastify wraps these. Use `verifyAndReceive()` manually instead.
- **Parsing JSON before HMAC verification:** The signature is computed on the raw string body. Parsing to JSON and re-stringifying changes whitespace/key order, breaking verification.
- **Global content type parser override:** Replacing Fastify's JSON parser globally would break all existing routes. Scope the raw body parser to the webhook route only.
- **Polling for reactions:** Reactions arrive via webhook events. Never poll for them.
- **One comment per update:** Creates noise. Always edit the single progress comment in-place.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HMAC-SHA256 verification | Custom crypto.createHmac + timingSafeEqual | `@octokit/webhooks.verify()` | Handles encoding, timing-safe comparison, error formatting |
| JWT generation for App auth | Manual JWT with jsonwebtoken | `@octokit/app` internal auth | Token refresh, expiry, installation token caching |
| Installation token management | Manual POST to /app/installations/:id/access_tokens | `app.getInstallationOctokit()` | Auto-refresh, caching, scoped permissions |
| Webhook event types | Manual interface definitions | `@octokit/webhooks-types` | Always up-to-date with GitHub API, 100+ event types |
| Collaborator permission check | Raw fetch to GitHub API | `octokit.rest.repos.getCollaboratorPermissionLevel()` | Typed response, error handling |
| Reaction creation | Raw fetch | `octokit.rest.reactions.createForIssueComment()` | Typed, handles media type headers |

**Key insight:** The Octokit ecosystem is comprehensive and well-typed. Every GitHub API operation has a corresponding typed method. Using raw fetch would require maintaining types, handling pagination, managing auth tokens, and dealing with rate limits manually.

## Common Pitfalls

### Pitfall 1: Raw Body Access in Fastify
**What goes wrong:** Fastify parses request bodies automatically. By the time your handler runs, `request.body` is a parsed object, not the raw string needed for HMAC verification.
**Why it happens:** Fastify's default JSON content type parser.
**How to avoid:** Use a scoped content type parser for the webhook route that preserves the raw string. Or use `@fastify/raw-body` plugin. Register the webhook route on a separate Fastify instance/prefix with its own parser.
**Warning signs:** Webhook signature verification always fails.

### Pitfall 2: Webhook Event Deduplication
**What goes wrong:** GitHub may redeliver webhooks (manual redeliver, network retry). Processing the same event twice could dispatch duplicate runs.
**Why it happens:** Webhook delivery is at-least-once, not exactly-once.
**How to avoid:** Track `x-github-delivery` IDs in a Set or database table. Skip already-processed delivery IDs. The dispatcher already deduplicates by issue ID (if run exists, skip), which provides a natural safety net.
**Warning signs:** Duplicate runs for the same issue.

### Pitfall 3: Race Between Webhook and Polling
**What goes wrong:** Both the webhook and the polling tick see the same new issue and try to dispatch simultaneously.
**Why it happens:** Webhooks are near-instant; polling runs on a timer. Both feed into `dispatchIssue()`.
**How to avoid:** `dispatchIssue()` already uses `claimIssue()` which checks `state.claimed` -- this is the deduplication point. Ensure webhook handler converts webhook payload to `TrackerIssue` format and calls the same dispatcher path. No additional locking needed.
**Warning signs:** Duplicate dispatch attempts in logs.

### Pitfall 4: Comment ID Tracking for Reactions
**What goes wrong:** A reaction event fires with a `comment_id` but there's no way to map it back to a forgectl run.
**Why it happens:** No storage linking bot comment IDs to run IDs.
**How to avoid:** When the bot creates a progress comment, store `{ commentId, runId, issueNumber, repo }` either in a new database table or as metadata on the run record. On reaction event, look up by comment ID.
**Warning signs:** Reactions silently ignored because no run mapping found.

### Pitfall 5: Clarification Reply Detection
**What goes wrong:** Any comment on an issue with a paused run is treated as the clarification reply, including bot comments or unrelated discussion.
**Why it happens:** Naive "any new comment resumes" logic.
**How to avoid:** Only resume when: (1) the run is in `waiting_for_input`, (2) the comment is a reply to the bot's clarification comment (check `in_reply_to` or thread), (3) the commenter is the issue author or a collaborator. If `in_reply_to` is not reliable, check that the comment is from the issue author and the run is waiting.
**Warning signs:** Premature resume from unrelated comments.

### Pitfall 6: GitHub App Private Key Loading
**What goes wrong:** Private key path in config but file doesn't exist, has wrong permissions, or has wrong format.
**Why it happens:** PKCS#1 vs PKCS#8 format differences, newline handling in PEM.
**How to avoid:** Validate at daemon startup: file exists, readable, starts with `-----BEGIN`. `@octokit/app` accepts the PEM string directly. Read file with `fs.readFileSync(path, 'utf-8')`.
**Warning signs:** JWT auth failures on startup, cryptic "invalid key" errors.

### Pitfall 7: Check Run Requires head_sha
**What goes wrong:** Can't create a check run without a commit SHA, but issue-triggered runs may not have a PR or commit yet.
**Why it happens:** Check runs are commit-scoped, not issue-scoped.
**How to avoid:** Only create check runs when there IS a PR (output mode = git). Get the head SHA from the PR's head branch. Create check run after PR is opened/updated, not at issue dispatch time.
**Warning signs:** API errors about missing head_sha.

function createGitHubApp(config: GitHubAppConfig): App {

### Collaborator Permission Check
```typescript
// Source: GitHub REST API docs
async function hasWriteAccess(
  octokit: Octokit,
  owner: string,
  repo: string,
  username: string,
): Promise<boolean> {
  try {
    const { data } = await octokit.rest.repos.getCollaboratorPermissionLevel({
      owner, repo, username,
    });
    return ['admin', 'write'].includes(data.permission);
  } catch {
    return false; // Not a collaborator or API error
  }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Probot framework | @octokit/app directly | 2023+ | No Express dependency, works with any HTTP framework |
| X-Hub-Signature (SHA-1) | X-Hub-Signature-256 (SHA-256) | 2021 | Must use SHA-256 header, SHA-1 deprecated |
| Manual JWT + token refresh | @octokit/app auto-manages | Always current | Token caching, refresh, installation scoping |
| reactions preview media type | Standard API (no preview header) | 2022 | No special Accept header needed for reactions |

**Deprecated/outdated:**
- `X-Hub-Signature` (SHA-1): Still sent by GitHub for backward compat, but use `X-Hub-Signature-256` exclusively
- Probot: Still maintained but brings Express baggage; not suitable when Fastify is the HTTP framework

## Open Questions

1. **Raw body scoping strategy**
   - What we know: Fastify needs raw body for webhook route only, parsed JSON for everything else
   - What's unclear: Whether `@fastify/raw-body` plugin or a route-scoped content type parser is cleaner
   - Recommendation: Use a route-scoped approach -- register webhook handler with custom content type parser on a Fastify prefix (`/api/v1/github/`) to isolate from existing routes

2. **Comment ID to Run ID storage**
   - What we know: Need to map bot comment IDs back to run IDs for reaction handling
   - What's unclear: New table vs. column on runs table vs. in-memory map
   - Recommendation: Add `githubCommentId` column to runs table (nullable). Simple, queryable, survives daemon restart

3. **Installation ID discovery**
   - What we know: @octokit/app can iterate installations, but config may specify one explicitly
   - What's unclear: Whether multi-installation support is needed in v2.0
   - Recommendation: Support single installation ID in config. If omitted, discover from first webhook event's `installation.id` field and cache it
