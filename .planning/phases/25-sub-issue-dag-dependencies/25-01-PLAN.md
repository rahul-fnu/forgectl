export interface SubIssueEntry {
  parentId: string;                    // Issue number string ("42")
  childIds: string[];                  // Issue number strings of direct children
  childStates: Map<string, string>;    // childId -> GitHub state ("open"/"closed")
  fetchedAt: number;                   // Date.now() at fetch time
}

export class SubIssueCache {
  private readonly ttlMs: number;
  private entries: Map<string, SubIssueEntry>;

  constructor(ttlMs?: number);         // Default 5 * 60 * 1000
  get(parentId: string): SubIssueEntry | null;
  set(entry: SubIssueEntry): void;
  invalidate(parentId: string): void;
  invalidateAll(): void;
  getAllEntries(): SubIssueEntry[];    // Returns non-expired entries
}
```

The `get()` method checks TTL on read and lazily deletes expired entries. `getAllEntries()` filters out expired entries (lazy cleanup) so the scheduler gets only fresh data for terminalIssueIds.

export interface IssueDAGNode {
  id: string;
  blocked_by: string[];
}

export function detectIssueCycles(issues: IssueDAGNode[]): string | null;
```

IMPORTANT: Do NOT use `validateDAG()` from `pipeline/dag.ts` directly because:
1. `PipelineDefinition` requires `name` and `PipelineNode` requires `task` -- these are meaningless for issue nodes.
2. `validateDAG()` errors on unknown node references (step 2 in its logic), but issue DAGs legitimately reference cross-repo issues that aren't in the local set.

Instead, implement a standalone DFS cycle detector specific to issue graphs. The algorithm is the same 3-color DFS from `pipeline/dag.ts` (WHITE/GRAY/BLACK), but:
- Only checks for cycles (skips duplicate ID check, unknown reference check, root node check)
- Operates on `IssueDAGNode[]` directly
- Filters `blocked_by` to only include IDs present in the input set (silently ignores external refs)
- Returns a human-readable cycle description string or null
