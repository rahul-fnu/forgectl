    - POST /api/v1/runs/:id/approve with run in pending_output_approval returns { status: "approved", runId, previousStatus: "pending_output_approval" }
    - POST /api/v1/runs/:id/approve with run not found returns 404 with { error: { code: "NOT_FOUND", message } }
    - POST /api/v1/runs/:id/approve with run in "running" returns 409 with { error: { code: "CONFLICT", message } }
    - POST /api/v1/runs/:id/reject with body { reason: "too risky" } transitions to rejected
    - POST /api/v1/runs/:id/reject with body { action: "revision_requested", feedback: "fix X" } stores feedback and transitions
    - POST /api/v1/runs/:id/reject without runRepo returns 503
  </behavior>
  <action>
    Add two new route handlers to `src/daemon/routes.ts`, following the exact pattern of the existing `/api/v1/runs/:id/resume` endpoint:

    1. **POST /api/v1/runs/:id/approve**:
       - Guard: if `!runRepo`, return 503 with standard NOT_CONFIGURED error
       - Call `approveRun(runRepo, runId)` from `src/governance/approval.ts`
       - On success, return `{ status: "approved", runId, previousStatus }`
       - Catch errors: "not found" -> 404, other -> 409 (CONFLICT)

    2. **POST /api/v1/runs/:id/reject**:
       - Guard: if `!runRepo`, return 503
       - Parse body: `{ reason?: string; action?: "reject" | "revision_requested"; feedback?: string }`
       - If `action === "revision_requested"` and `feedback` is provided: call `requestRevision(runRepo, runId, feedback)`
       - Otherwise: call `rejectRun(runRepo, runId, reason)`
       - On success, return `{ status: body.action ?? "rejected", runId }`
       - Catch errors: "not found" -> 404, other -> 409

    Import `approveRun`, `rejectRun`, `requestRevision` from `../governance/approval.js`.

    Write tests in `test/unit/governance-routes.test.ts` using the same test pattern as existing route tests — create a minimal Fastify instance, register routes with a mock runRepo, and test each endpoint with `app.inject()`.

    1. **Pre-gate in dispatcher.ts** (`dispatchIssue` function):

       Add the pre-approval gate check BEFORE the `void executeWorkerAndHandle(...)` call:

       ```typescript
       import { needsPreApproval } from "../governance/autonomy.js";
       import { enterPendingApproval } from "../governance/approval.js";
       import { evaluateAutoApprove } from "../governance/rules.js";
       import type { AutoApproveContext } from "../governance/types.js";
       ```

       After `claimIssue` succeeds, before `void executeWorkerAndHandle(...)`:
       - Get autonomy level from `config` (it's in the merged workflow config). Access it via the workflow definition on the resolved config: check if `config` has a workflow with autonomy set. The autonomy field is on the WorkflowDefinition (from WorkflowSchema). The dispatcher receives `config: ForgectlConfig` which doesn't directly have autonomy — the autonomy is on the per-workflow definition. Since the dispatcher doesn't currently have the resolved WorkflowDefinition, add an optional `autonomy` parameter to `dispatchIssue` (default "full" for backward compat), or resolve it from the workflow config inside `executeWorker`. The simplest approach: pass a `resolvedWorkflow?: WorkflowDefinition` parameter. But to minimize signature changes, read autonomy from the workflow config that `executeWorker` would resolve. Since `executeWorker` resolves the workflow internally, extract the autonomy check into the worker flow. **Actually, the cleanest approach per the research**: check autonomy INSIDE `executeWorkerAndHandle` at the very top, before any container setup. If pre-approval needed, call `enterPendingApproval` and return early (the REST endpoint will re-trigger dispatch on approve).

       Concretely, in `executeWorkerAndHandle`:
       - After creating the WorkerInfo in `state.running`, resolve the workflow to get autonomy level
       - If `needsPreApproval(autonomy)`:
         - Build auto-approve context: `{ labels: issue.labels, workflowName: workflowName }`
         - If `workflow.auto_approve` exists and `evaluateAutoApprove(workflow.auto_approve, autoApproveCtx)` returns true: skip gate, continue execution
         - Otherwise: emit `approval_required` event, set run status to `pending_approval` (if using runRepo), and return early without executing the worker
       - If no pre-approval needed: continue with existing flow

       **Note**: The dispatcher currently does not use RunRepository (it's orchestrator-mode, not daemon-mode). For orchestrator-mode runs, the pre-gate needs to persist the run state. Check if a `runRepo` or `DurabilityDeps` is available. If not available (e.g., orchestrator without storage), log a warning and skip the gate (or make runRepo available via dependency injection). The simplest approach: add optional `runRepo?: RunRepository` to the parameters. If runRepo is unavailable and pre-approval is needed, log warning and proceed without gate.

    2. **Post-gate in single.ts** (`executeSingleAgent` function):

       Read `src/orchestration/single.ts` to understand the execution flow. After validation passes and before output collection:
       - Check `needsPostApproval(autonomy)` where autonomy comes from the workflow config
       - If post-approval needed:
         - Build auto-approve context with actual cost: `{ labels, workflowName, actualCost }`
         - If auto-approve passes: skip gate, proceed to output collection
         - Otherwise: collect output to the host directory (git branch created, files written) but call `enterPendingOutputApproval(runRepo, runId)` and return early. The output is on the host — container can be cleaned up. On approve, the status simply transitions to completed.
       - If no post-approval needed: continue with existing flow

       **Important per pitfall #4**: Output must be collected BEFORE entering pending_output_approval. The container cleanup in the finally block is fine because output is already on the host.

    3. Verify backward compatibility: when autonomy is "full" (the default), both `needsPreApproval` and `needsPostApproval` return false, so the existing flow is completely unchanged.
