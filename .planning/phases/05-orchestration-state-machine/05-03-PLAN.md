  <name>Task 1: Dispatcher — candidate filtering, priority sorting, dispatch</name>
  <files>src/orchestrator/dispatcher.ts, test/unit/orchestrator-dispatcher.test.ts</files>
  <behavior>
    - filterCandidates excludes issues already in claimed Set
    - filterCandidates excludes issues already in running Map
    - filterCandidates excludes issues with blocked_by IDs that are not in terminal states (pass terminalIds Set)
    - filterCandidates returns only issues fitting available slots
    - sortCandidates sorts by priority ascending (P0 before P1 before P2)
    - sortCandidates uses created_at as tiebreaker (oldest first)
    - sortCandidates uses identifier as final tiebreaker
    - sortCandidates handles null priority (treated as lowest/infinity)
    - dispatchIssue claims the issue, starts worker in background, adds to running Map
    - dispatchIssue updates tracker labels with in_progress_label (best-effort, failure doesn't block)
    - dispatchIssue on worker complete: posts comment, classifies failure, schedules retry or releases
    - dispatchIssue on max retries exhausted: releases issue, posts failure comment, removes in_progress label
  </behavior>
  <action>
Create `src/orchestrator/dispatcher.ts` with:

1. `filterCandidates(candidates: TrackerIssue[], state: OrchestratorState, terminalIssueIds: Set<string>): TrackerIssue[]`
   - Filter out issues where `state.claimed.has(issue.id)` is true
   - Filter out issues where `state.running.has(issue.id)` is true
   - Filter out issues where any `issue.blocked_by` ID is NOT in `terminalIssueIds` (i.e., blocker is still active)
   - Return filtered list

2. `sortCandidates(issues: TrackerIssue[]): TrackerIssue[]`
   - Sort by extracted priority number ascending (use `extractPriorityNumber` — parse from `priority` field or labels with P0/P1/P2 or priority:high/medium/low pattern). Null priority = Infinity
   - Tiebreak by `created_at` ascending (oldest first)
   - Final tiebreak by `identifier` alphabetically
   - Note: Priority extraction logic already exists in `src/tracker/github.ts` but is not exported. Re-implement a simple version here: parse P0/P1/P2/P3/P4 from labels, or priority:critical=0/high=1/medium=2/low=3. If priority field is set directly, parse its numeric value.

3. `dispatchIssue(issue, state, tracker, config, workspaceManager, promptTemplate, logger): void`
   - Call `claimIssue(state, issue.id)` — if returns false (already claimed), return
   - Best-effort: `tracker.updateLabels(issue.id, [config.orchestrator.in_progress_label], []).catch(err => logger.warn(...))`
   - Start `executeWorker()` as fire-and-forget (`void executeWorkerAndHandle(...)`)
   - In the async handler:
     a. Create `onActivity` callback that updates `worker.lastActivityAt = Date.now()`
     b. Add WorkerInfo to `state.running` with current timestamps and attempt
     c. Await `executeWorker(issue, config, workspaceManager, promptTemplate, attempt, logger, onActivity)`
     d. Remove from `state.running`
     e. Post comment via `tracker.postComment(issue.id, result.comment).catch(...)`
     f. Classify failure: if "continuation" → scheduleRetry with continuation_delay_ms
     g. If "error": increment retryAttempts, check max_retries. If exhausted → releaseIssue + post failure comment + remove in_progress label. Otherwise → scheduleRetry with calculateBackoff delay
     h. Retry callback: re-fetch candidates, check if issue still eligible, dispatch or release

Mock all external dependencies in tests.

<task type="auto" tdd="true">
  <name>Task 2: Reconciler and scheduler with tests</name>
  <files>src/orchestrator/reconciler.ts, src/orchestrator/scheduler.ts, test/unit/orchestrator-reconciler.test.ts, test/unit/orchestrator-scheduler.test.ts</files>
  <behavior>
    - reconcile: fetches current states for all running issue IDs via tracker.fetchIssueStatesByIds
    - reconcile: terminal state issue -> calls session.close(), cleanupRun(), workspaceManager.remove(), releaseIssue()
    - reconcile: non-active/non-terminal state -> calls session.close(), cleanupRun(), releaseIssue() (no workspace removal)
    - reconcile: active state -> updates issue snapshot in WorkerInfo
    - reconcile: fetchIssueStatesByIds failure -> logs warning, does NOT stop any workers
    - reconcile: stall detection -> if Date.now() - lastActivityAt > stallTimeoutMs -> kills session, schedules error retry
    - scheduler: startScheduler returns stop function
    - scheduler: tick runs reconcile before dispatch
    - scheduler: uses setTimeout chain (not setInterval)
    - scheduler: stopped flag prevents further ticks
  </behavior>
  <action>
1. Create `src/orchestrator/reconciler.ts` with:

   `reconcile(state, tracker, workspaceManager, config, logger): Promise<void>`
   - If `state.running.size === 0`, return early
   - Get all running issue IDs: `[...state.running.keys()]`
   - Try `tracker.fetchIssueStatesByIds(ids)` — on failure, log warning and return (keep all workers running)
   - For each running worker, check fetched state:
     a. If state is in `config.tracker.terminal_states`: close session, cleanup container, remove workspace, release issue
     b. If state is NOT in `config.tracker.active_states` AND NOT terminal: close session, cleanup container, release issue (no workspace removal)
     c. If state is in `config.tracker.active_states`: update `worker.issue.state` in-memory
   - Stall detection (separate loop): for each running worker, if `Date.now() - worker.lastActivityAt > config.orchestrator.stall_timeout_ms`:
     a. Log stall warning with issue identifier and elapsed time
     b. Close session, cleanup container
     c. Remove from running Map
     d. Increment retryAttempts, schedule error retry with backoff (if retries remain), or release

   Catch errors per-worker in reconciliation (one worker's cleanup failure shouldn't prevent others).

2. Create `src/orchestrator/scheduler.ts` with:

   `TickDeps` interface: `{ state, tracker, workspaceManager, slotManager, config, promptTemplate, logger }`

   `tick(deps: TickDeps): Promise<void>`
   - Step 1: await `reconcile(deps.state, deps.tracker, deps.workspaceManager, deps.config, deps.logger)`
   - Step 2: validate config (check tracker is defined — if not, skip dispatch with warning)
   - Step 3: `const candidates = await deps.tracker.fetchCandidateIssues()`
   - Step 4: `const eligible = filterCandidates(candidates, deps.state, terminalIds)` — terminalIds from recent reconciliation or empty Set
   - Step 5: `const sorted = sortCandidates(eligible)`
   - Step 6: `const available = deps.slotManager.availableSlots(deps.state.running)`
   - Step 7: for each issue in `sorted.slice(0, available)`, call `dispatchIssue(...)`

   `startScheduler(deps: TickDeps): () => void`
   - Sets `stopped = false`
   - Calls `scheduleTick()` which: runs `tick()`, catches errors (log, don't crash), if not stopped schedules next tick via `setTimeout(scheduleTick, deps.config.orchestrator.poll_interval_ms)`
   - Returns a stop function that sets `stopped = true` and clears the pending tick timer

   Use `vi.useFakeTimers()` in tests to control tick timing.
