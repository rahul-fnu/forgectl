  <behavior>
    - synthesize() invokes the agent (via executeWorkerFn) with a synthesis prompt containing all child outcomes
    - buildSynthesisPrompt includes original issue title, per-child status badge and output, and instructions for structured markdown
    - synthesize() returns the agent's stdout as the synthesis comment text
    - synthesize() runs even when some children permanently failed — partial results are included
    - If synthesis agent invocation fails, returns a fallback summary: "Delegation for '{issue.title}' completed. {N} children: {M} succeeded, {K} failed."
    - Synthesis comment is a single markdown block with overall status header, per-child sections, and lead synthesis paragraph
  </behavior>
  <action>
    1. In `src/orchestrator/delegation.ts`, replace the stub `synthesize()` implementation from Plan 02 with the full version.

    2. Implement `buildSynthesisPrompt(parentIssue: TrackerIssue, outcomes: ChildOutcome[]): string`:
       - Start with: `You previously decomposed issue "{parentIssue.title}" into subtasks.`
       - Add: `All subtasks have now completed. Here are the results:`
       - For each outcome, add a section:
         ```
         ### Subtask {spec.id} — {COMPLETED|FAILED}
         {stdout or errorMessage or "(no output)"}
         ```
       - End with instructions: "Produce a structured markdown summary with: 1. An overall status header (all passed / partial failure / all failed) 2. A one-line status badge per subtask 3. A synthesis paragraph summarizing what was accomplished and what (if anything) needs follow-up"

    3. Implement `synthesize(parentIssue, outcomes)`:
       - Create a synthetic issue for the synthesis call: use `toSyntheticIssue({ id: 'synthesis', task: buildSynthesisPrompt(parentIssue, outcomes) }, parentIssue)`
       - Call `executeWorkerFn(syntheticIssue, config, wsManager, synthesisPrompt, 1, logger)`
       - Return `result.agentResult.stdout` as the synthesis text
       - On error: return fallback summary (count completed vs failed)

    4. Wire synthesis into `runDelegation` (from Plan 02):
       - After all children settle (Promise.allSettled resolves), call `this.synthesize(parentIssue, outcomes)`
       - Post the synthesis comment to the tracker: `tracker.postComment(parentIssue.id, synthesisComment)` — this requires adding `tracker: TrackerAdapter` to DelegationDeps
       - Return the DelegationOutcome with all child outcomes

    5. Add/extend tests in `test/unit/delegation-manager.test.ts`:
       - Test: synthesize() invokes agent with correct prompt content
       - Test: synthesis runs when some children failed (partial results)
       - Test: synthesis fallback when agent invocation throws
       - Test: buildSynthesisPrompt includes all child outcomes with status badges
       - Test: single postComment call after synthesis (not per-child)

<task type="auto" tdd="true">
  <name>Task 2: Daemon restart recovery for in-flight delegations</name>
  <files>src/orchestrator/reconciler.ts, src/orchestrator/index.ts, test/unit/orchestrator-reconciler.test.ts</files>
  <behavior>
    - On daemon restart, query delegations table for rows with status 'pending' or 'running'
    - Group in-flight delegations by parentRunId
    - For each parent with in-flight children: mark stale 'running' delegations as 'failed' (they were interrupted)
    - For each parent with in-flight 'pending' delegations: re-dispatch those children
    - If all delegations for a parent are now terminal (completed/failed), trigger synthesis
    - Recovery is non-fatal: errors are caught and logged as warnings
  </behavior>
  <action>
    1. In `src/orchestrator/reconciler.ts`, add a new exported function `recoverDelegations`:
       ```typescript
       export async function recoverDelegations(
         delegationRepo: DelegationRepository,
         logger: Logger,
       ): Promise<{ recovered: number; failed: number }> {
       ```
       - Query `delegationRepo.list()` and filter for non-terminal statuses (pending, running)
       - For 'running' rows: they were mid-execution when daemon crashed. Mark as 'failed' with lastError "daemon restart — execution interrupted"
       - For 'pending' rows: they were queued but never started. Leave as 'pending' for the DelegationManager to pick up.
       - Group by parentRunId and log summary: "Delegation recovery: {N} in-flight delegations for {M} parents. {K} marked failed (interrupted), {L} pending for re-dispatch."
       - Return counts for telemetry

    2. In `src/orchestrator/index.ts`, extend `startupRecovery()`:
       - If a DelegationRepository is available (add optional delegationRepo to OrchestratorOptions), call `recoverDelegations(delegationRepo, logger)` during startup
       - Log the recovery results
       - This is best-effort: wrap in try/catch and log warnings on failure

    3. Add tests in `test/unit/orchestrator-reconciler.test.ts`:
       - Test: running delegations marked as failed on recovery
       - Test: pending delegations left as-is
       - Test: recovery with no in-flight delegations is a no-op
       - Test: recovery errors are caught (non-fatal)

    4. Ensure the reconciler does NOT re-dispatch children directly. It only updates statuses. Re-dispatch is the DelegationManager's responsibility on the next scheduler tick when it detects pending delegations for a parent. This keeps the reconciler simple and avoids circular dependencies.
  </action>

  </verify>
