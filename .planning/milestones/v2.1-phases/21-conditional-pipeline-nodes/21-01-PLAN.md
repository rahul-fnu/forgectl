  <name>Task 1: Create condition.ts evaluator and shorthand expansion</name>
  <files>src/pipeline/condition.ts, test/unit/pipeline-condition.test.ts</files>
  <behavior>
    - evaluateCondition('build == "completed"', { build: "completed" }) returns true
    - evaluateCondition('build == "completed"', { build: "failed" }) returns false
    - evaluateCondition('build == "completed" and test == "completed"', { build: "completed", test: "completed" }) returns true
    - evaluateCondition('not (build == "failed")', { build: "completed" }) returns true
    - evaluateCondition with syntax error throws ConditionSyntaxError
    - evaluateCondition referencing unknown variable throws ConditionVariableError
    - expandShorthands converts if_failed: "test" to condition: 'test == "failed"' and adds "test" to depends_on
    - expandShorthands converts if_passed: "build" to condition: 'build == "completed"' and adds "build" to depends_on
    - expandShorthands does not duplicate "test" in depends_on if already present
    - expandShorthands throws when both condition and if_failed are set on same node
    - expandShorthands leaves nodes without shorthands unchanged
  </behavior>
  <action>
    Create `src/pipeline/condition.ts` with:

    1. **Types:** `NodeStatusContext = Record<string, "completed" | "failed" | "skipped">`. Export error classes `ConditionSyntaxError` and `ConditionVariableError` (both extend Error).

    2. **`evaluateCondition(expression: string, context: NodeStatusContext): boolean`** — Wraps filtrex `compileExpression`. Use `customProp` option to intercept unknown variable access and throw `ConditionVariableError`. Catch filtrex compile errors and wrap as `ConditionSyntaxError`. Always `return Boolean(result)` since filtrex returns numeric 1/0 for boolean operations.

    3. **`expandShorthands(pipeline: PipelineDefinition): PipelineDefinition`** — Post-parse transform. For each node:
       - If both `condition` and (`if_failed` or `if_passed`) exist, throw error (mutually exclusive per user decision).
       - If `if_failed: targetId`, set `condition: '${targetId} == "failed"'`, auto-add targetId to depends_on if not present, clear if_failed.
       - If `if_passed: targetId`, set `condition: '${targetId} == "completed"'`, auto-add targetId to depends_on if not present, clear if_passed.
       - Build new node objects (do not mutate originals — Zod objects may be frozen).
       - Return new PipelineDefinition with expanded nodes.

    Create `test/unit/pipeline-condition.test.ts` with tests for all behaviors listed above. Write tests FIRST (RED), then implement (GREEN).
  </action>

  </verify>
  <done>evaluateCondition handles truthy/falsy/error cases correctly; expandShorthands transforms if_failed/if_passed to condition strings and auto-adds depends_on; all tests green</done>
</task>
