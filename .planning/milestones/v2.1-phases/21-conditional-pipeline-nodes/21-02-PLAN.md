<task type="auto">
  <name>Task 1: Replace static topo-sort loop with ready-queue executor and wire condition evaluation</name>
  <files>src/pipeline/executor.ts, test/unit/pipeline-executor.test.ts</files>
  <action>
    **Refactor `execute()` method in PipelineExecutor** — replace the static `for (const nodeId of order)` loop (lines 113-151) with a ready-queue drain loop. Keep all code before (DAG validation, dry-run check, prepareRerunSelection, hydration) and after (status determination, event emission, return) unchanged.

    The new execution core:

    1. **Build dependents map** using `buildDependentsMap(this.pipeline)` imported from dag.ts.

    2. **Seed ready queue** — a `Set<string>` of node IDs. Initial seeds: nodes with no dependencies that are in `selection.executeNodes` and not already in a terminal state.

    3. **`processNode(nodeId)` async function:**
       - Build upstream status context: for each node in `node.depends_on`, look up `this.nodeStates.get(depId).status` and build a `NodeStatusContext` object.
       - If node has a `condition` field, call `evaluateCondition(node.condition, context)`:
         - If `false` and `node.else_node` is set: mark node as skipped with `skipReason: 'condition false: ${node.condition}'`, then try to enqueue `else_node` (if all its deps are terminal and it's in executeNodes).
         - If `false` and no `else_node`: mark node as skipped with same skipReason, then cascade-skip all downstream dependents.
         - If `true`: proceed to `this.executeNode(node)` as before.
         - If evaluation throws (ConditionSyntaxError or ConditionVariableError): set pipeline status to failed, set node error, do NOT continue processing.
       - If node has NO `condition`: execute it directly via `this.executeNode(node)`.
       - After terminal state: delete from inFlight, then for each dependent in dependentsMap, check if all its deps are terminal — if so, add to readyQueue.

    4. **Cascade skip helper** `propagateCascadeSkip(nodeId, dependentsMap, nodeMap, selection)`:
       - For each downstream dependent of nodeId (via dependentsMap), if the dependent is in `selection.executeNodes` and not already terminal:
         - Mark as skipped with `skipReason: 'dependency ${nodeId} was skipped'`
         - Recursively cascade to its dependents
       - Do NOT cascade-skip through `else_node` activation — else_node is an alternative path.

    5. **isNodeReady(nodeId) check:**
       - All `depends_on` are in terminal state (completed, failed, skipped)
       - Node is in `selection.executeNodes`
       - Node is not already in a terminal status
       - Node is not already in `inFlight`

    6. **Drain loop:**
       ```
       while (readyQueue.size > 0 || inFlight.size > 0) {
         for (const nodeId of [...readyQueue]) {
           if (inFlight.size >= maxParallel) break;
           readyQueue.delete(nodeId);
           const promise = processNode(nodeId);
           inFlight.set(nodeId, promise);
         }
         if (inFlight.size > 0) await Promise.race(inFlight.values());
       }
       ```

    7. **Cascade skip vs checkpoint-hydrated skip:** When checking if a dependency's skip should cascade, check `hydratedFromCheckpoint` — if the dependency was hydrated from checkpoint (has result with output), do NOT cascade skip. Only cascade when `status === "skipped"` AND `hydratedFromCheckpoint` is falsy.

    8. **Fatal condition errors:** If `evaluateCondition` throws, catch the error, mark the node as failed with the error message, and set `pipelineStatus = "failed"`. Do NOT continue processing other nodes — break out of the drain loop. This implements COND-06.

    **Extend `test/unit/pipeline-executor.test.ts`** with new test cases:
    - Pipeline with condition that evaluates true: node executes normally
    - Pipeline with condition that evaluates false: node is skipped with correct skipReason
    - Pipeline with condition false + else_node: conditional node skipped, else_node executes
    - Pipeline with cascade skip: A -> B(condition:false) -> C: both B and C skipped
    - Pipeline with unknown variable in condition: pipeline fails immediately (COND-06)
    - Existing tests (linear, parallel, fan-in, checkpoint) still pass with ready-queue
