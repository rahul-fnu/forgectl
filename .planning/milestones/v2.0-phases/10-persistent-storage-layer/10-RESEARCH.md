# Phase 10: Persistent Storage Layer - Research

**Researched:** 2026-03-09
**Domain:** SQLite + Drizzle ORM embedded database for Node.js daemon
**Confidence:** HIGH

## Summary

Phase 10 introduces SQLite as the persistent storage backend for the forgectl daemon, replacing the current in-memory data structures (RunQueue array, PipelineRunService Map, BoardStore JSON files, OrchestratorState Sets/Maps). The stack is locked: Drizzle ORM with better-sqlite3 driver, SQLite dialect, programmatic migrations on startup.

The current daemon stores all state ephemerally: `RunQueue` uses an in-memory array, `PipelineRunService` uses a `Map<string, PipelineRunEntry>`, `BoardStore` uses JSON files with filesystem locks, and `OrchestratorState` uses `Set`/`Map` collections. All of this is lost on daemon restart. Phase 10 creates the foundation layer -- database initialization, schema, migrations, and typed repository functions -- that later phases (11-16) build on.

**Primary recommendation:** Use Drizzle ORM with better-sqlite3 driver, `drizzle-orm/better-sqlite3/migrator` for file-based migrations that run automatically on daemon startup, and a repository pattern with typed query/mutation functions per entity. Do NOT use the programmatic `pushSQLiteSchema` API (it is underdocumented and designed for dev prototyping, not production startup).

## Architecture Patterns

### Recommended Project Structure
```
src/
├── storage/
│   ├── database.ts          # Database singleton: init, connection, WAL mode, close
│   ├── schema.ts            # All Drizzle table definitions (runs, pipelines, board_states)
│   ├── migrator.ts          # Wraps migrate() call for daemon startup
│   └── repositories/
│       ├── runs.ts           # CRUD for queued/running/completed runs
│       ├── pipelines.ts      # CRUD for pipeline runs and node states
│       └── boards.ts         # CRUD for board state (replaces JSON file store)
```

### Pattern 1: Database Singleton
**What:** Single module that creates and exports the database connection, called once during daemon startup.
**When to use:** Always -- SQLite connections should be shared (single-writer).
**Example:**
```typescript
// src/storage/database.ts
import Database from "better-sqlite3";
import { drizzle } from "drizzle-orm/better-sqlite3";
import { join } from "node:path";
import { mkdirSync } from "node:fs";
import * as schema from "./schema.js";

export type AppDatabase = ReturnType<typeof createDatabase>;

export function createDatabase(dbPath?: string): ReturnType<typeof drizzle> {
  const resolvedPath = dbPath ?? join(
    process.env.HOME || "/tmp",
    ".forgectl",
    "forgectl.db"
  );
  mkdirSync(dirname(resolvedPath), { recursive: true });

  const sqlite = new Database(resolvedPath);
  sqlite.pragma("journal_mode = WAL");
  sqlite.pragma("foreign_keys = ON");
  sqlite.pragma("busy_timeout = 5000");

  return drizzle(sqlite, { schema });
}
```

### Pattern 2: File-Based Migrations on Startup
**What:** Use `drizzle-kit generate` to create SQL migration files in a `drizzle/` folder, then call `migrate()` at daemon startup before accepting requests.
**When to use:** Always for production. Migration files are committed to git, providing a clear history.
**Example:**
```typescript
// src/storage/migrator.ts
import { migrate } from "drizzle-orm/better-sqlite3/migrator";
import type { AppDatabase } from "./database.js";

export function runMigrations(db: AppDatabase): void {
  // Drizzle migrate() is synchronous for better-sqlite3
  migrate(db, { migrationsFolder: "./drizzle" });
}
```

### Pattern 3: Typed Repository Functions
**What:** Each entity gets a repository module exporting typed async functions. Business logic calls repository functions, never constructs SQL or uses the db instance directly.
**When to use:** All database access in the codebase.
**Example:**
```typescript
// src/storage/repositories/runs.ts
import { eq } from "drizzle-orm";
import { runs } from "../schema.js";
import type { AppDatabase } from "../database.js";

export interface RunRepository {
  insert(run: typeof runs.$inferInsert): typeof runs.$inferSelect;
  findById(id: string): typeof runs.$inferSelect | undefined;
  updateStatus(id: string, status: string, completedAt?: string): void;
  list(): Array<typeof runs.$inferSelect>;
}

export function createRunRepository(db: AppDatabase): RunRepository {
  return {
    insert(run) {
      return db.insert(runs).values(run).returning().get();
    },
    findById(id) {
      return db.select().from(runs).where(eq(runs.id, id)).get();
    },
    updateStatus(id, status, completedAt) {
      db.update(runs)
        .set({ status, completedAt })
        .where(eq(runs.id, id))
        .run();
    },
    list() {
      return db.select().from(runs).all();
    },
  };
}
```

### Pattern 4: Schema Declaration with sqlite-core
**What:** Define tables using Drizzle's `sqliteTable` function with typed column builders.
**Example:**
```typescript
// src/storage/schema.ts
import { sqliteTable, text, integer } from "drizzle-orm/sqlite-core";

export const runs = sqliteTable("runs", {
  id: text("id").primaryKey(),
  task: text("task").notNull(),
  workflow: text("workflow"),
  status: text("status").notNull().default("queued"),
  options: text("options", { mode: "json" }),  // JSON column for CLIOptions
  submittedAt: text("submitted_at").notNull(),
  startedAt: text("started_at"),
  completedAt: text("completed_at"),
  result: text("result", { mode: "json" }),    // JSON column for ExecutionResult
  error: text("error"),
});

export const pipelineRuns = sqliteTable("pipeline_runs", {
  id: text("id").primaryKey(),
  pipelineDefinition: text("pipeline_definition", { mode: "json" }).notNull(),
  status: text("status").notNull().default("running"),
  nodeStates: text("node_states", { mode: "json" }),
  startedAt: text("started_at").notNull(),
  completedAt: text("completed_at"),
});
```

### Anti-Patterns to Avoid
- **Raw SQL in business logic:** Never use `db.run("SELECT ...")` in daemon routes or services. Always go through repository functions.
- **Multiple database connections:** SQLite is single-writer. Create one connection at startup, share it via dependency injection.
- **Storing timestamps as integers:** Use ISO 8601 text strings (the project convention per existing code).
- **Using `pushSQLiteSchema` for production migrations:** It has no migration history, no rollback story, and is underdocumented.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Schema migrations | Custom SQL version tracker | `drizzle-kit generate` + `migrate()` | Migration ordering, conflict detection, journal tracking already solved |
| SQL query building | String template SQL | Drizzle query builder (`db.select().from()...`) | SQL injection safety, type inference, refactor safety |
| Database file locking | Filesystem lock files | SQLite WAL mode + `busy_timeout` pragma | SQLite handles concurrent access natively; current BoardStore lock pattern is fragile |
| JSON serialization in columns | Custom serialize/deserialize | Drizzle `{ mode: "json" }` column type | Automatic JSON parse/stringify with type safety |
| Connection pooling | Pool manager | Nothing -- not needed | better-sqlite3 is synchronous, single connection is correct for embedded SQLite |

**Key insight:** SQLite with WAL mode and busy_timeout handles all concurrency the daemon needs. The current filesystem-based locking in BoardStore (using `openSync("wx")` lock files) can be entirely replaced by SQLite's native transaction isolation.

## Common Pitfalls

### Pitfall 1: Native Module Build Failures
**What goes wrong:** `better-sqlite3` is a native C++ addon. It requires a C++ compiler and `node-gyp` to install.
**Why it happens:** CI environments or Docker images may lack build tools.
**How to avoid:** Ensure `build-essential` (Linux) or equivalent is available. Add a note to CI config. The project already uses Docker extensively, so this is likely covered.
**Warning signs:** `npm install` fails with `gyp ERR!` messages.

### Pitfall 2: ESM Import of better-sqlite3
**What goes wrong:** `better-sqlite3` is a CommonJS module. Importing it in an ESM project requires care.
**Why it happens:** The project uses `"type": "module"` and ESM throughout.
**How to avoid:** Use `import Database from "better-sqlite3"` -- this works with `esModuleInterop: true` (already set in tsconfig). The `drizzle-orm/better-sqlite3` wrapper handles this internally.
**Warning signs:** `ERR_REQUIRE_ESM` or `default is not a constructor` errors.

### Pitfall 3: tsup Bundling Native Modules
**What goes wrong:** tsup tries to bundle `better-sqlite3` into the output, which fails because it is a native addon.
**Why it happens:** tsup bundles all dependencies by default.
**How to avoid:** Add `better-sqlite3` to the `external` array in `tsup.config.ts`:
```typescript
external: ["better-sqlite3"]
```
**Warning signs:** Build errors about `.node` files or missing native bindings.

### Pitfall 4: Synchronous vs Async API Confusion
**What goes wrong:** Developer wraps better-sqlite3 calls in unnecessary `async/await` or expects Promises.
**Why it happens:** better-sqlite3 is synchronous (unlike most DB drivers). Drizzle's better-sqlite3 adapter returns results directly, not Promises.
**How to avoid:** Repository functions for better-sqlite3 drizzle are synchronous. Methods like `.get()`, `.all()`, `.run()` return values directly. If the project convention requires async interfaces (for future driver swaps), wrap in `Promise.resolve()`.
**Warning signs:** `await` on a non-Promise value (TypeScript will warn).

### Pitfall 5: Missing WAL Mode
**What goes wrong:** Default SQLite journal mode (DELETE) blocks all readers during writes, causing daemon API latency spikes.
**Why it happens:** WAL mode must be explicitly enabled.
**How to avoid:** Set `pragma("journal_mode = WAL")` immediately after opening the connection.
**Warning signs:** Slow API responses when runs are being updated.

### Pitfall 6: Migration Folder Path in Bundled Builds
**What goes wrong:** `migrate()` looks for a `./drizzle/` folder relative to CWD, but the daemon may run from a different directory.
**Why it happens:** Migration folder path is relative.
**How to avoid:** Use `path.join(import.meta.dirname, "..", "drizzle")` or embed the migration folder path based on the package installation location. Alternatively, ship migrations alongside dist/.
**Warning signs:** "No migrations found" error on startup.

## Open Questions

1. **BoardStore migration strategy**
   - What we know: BoardStore currently uses JSON files with filesystem locks. SQLite replaces this entirely.
   - What's unclear: Whether to provide a one-time migration script for existing board JSON data, or start fresh (no users in production yet since v2.0 is new).
   - Recommendation: Start fresh. Board data is regenerated from definition files. No migration of existing JSON state needed.

2. **Migration file distribution in npm package**
   - What we know: `migrate()` needs access to the SQL migration folder at runtime.
   - What's unclear: Best way to ship migration files when forgectl is installed globally via npm.
   - Recommendation: Copy `drizzle/` folder into `dist/drizzle/` during build (extend `tsup.config.ts` `onSuccess`). Resolve path via `import.meta.dirname`.

3. **Repository interface: sync or async?**
   - What we know: better-sqlite3 is synchronous. Drizzle's better-sqlite3 adapter returns values directly.
   - What's unclear: Whether to make repository interfaces async (for potential future driver swap to libsql) or keep sync.
   - Recommendation: Keep synchronous for now. The project can add an async wrapper layer if/when switching drivers. YAGNI.
