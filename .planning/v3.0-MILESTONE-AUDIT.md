## Critical Integration Gaps

### Bug 1: Dual SubIssueCache Instances (SUBISSUE-03, SUBISSUE-05, SUBISSUE-06)

**Root cause:** `src/daemon/server.ts:117` creates `new SubIssueCache()` (Instance B) and passes it to Orchestrator and webhook handlers. But `src/tracker/github.ts:139` independently creates `new SubIssueCache()` (Instance A) inside the `createGitHubAdapter()` closure.

- **Instance A** (adapter): populated during `fetchCandidateIssues()` — this is where sub-issue data lives
- **Instance B** (orchestrator/webhooks): what the scheduler reads — **always empty**
- `scheduler.ts tick()` calls `deps.subIssueCache.getAllEntries()` on Instance B → `terminalIssueIds` is always an empty Set
- Webhook invalidation targets Instance B → no-op since it has no entries

**Fix:** Pass the server-created SubIssueCache into `createGitHubAdapter()` via the adapter factory, or expose the adapter's internal cache.

### Bug 2: githubContext Undefined in Polling Path (SUBISSUE-05, SUBISSUE-06)

**Root cause:** `src/orchestrator/dispatcher.ts:437` guards `triggerParentRollup` with `if (subIssueCache && githubContext)`. The scheduler at `src/orchestrator/scheduler.ts:94` passes `undefined` as `githubContext` (position 10).

- Polling-dispatched issues (primary production path) **never** have `githubContext`
- Only webhook-triggered dispatches provide `githubContext`
- Progress rollup and auto-close are dead code for all polling-originated work

**Fix:** Either decouple rollup from `githubContext` (construct Octokit from stored credentials), or provide an Octokit instance in the scheduler's dispatch path.
